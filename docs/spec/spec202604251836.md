# 中心化局域网保活系统需求文档

## 1. 核心目标
在学校机房环境下，建立一套去中心化的保活网络。
* **强制存活：** 确保局域网内处于活跃（或伪活跃）状态的机器数量不低于预设值 $N$。
* **隐形运行：** 拦截系统关机指令，与网络协商决定是否执行关机。用户无法感知程序存在，若拦截关机，则通过注销+黑屏欺骗用户，并卸载所有输入设备。如果用户尝试开机，则进行重启（调用底层API直接断电并开机）。
* **去中心化决策：** 无需中央服务器，所有节点平等，通过 P2P 和广播协议达成共识。

---

## 2. 系统组成与部署
### 2.1 结构划分
* **Sentinel Service (Logic Side):** * 采用 **Python** 编写，使用 `Nuitka` 编译为单文件 EXE 以实现反编译保护。
    * 以 `SYSTEM` 权限注册为隐蔽的 Windows 服务。
    * 负责网络协商、WoL 唤醒、逻辑决策。
* **Sentinel Hook DLL (C++ Side):**
    * 采用 **C++** 编写，利用 `MinHook` 框架。
    * 由 Python 服务注入至 `winlogon.exe`。
    * 负责底层 API 拦截及黑屏视觉模拟。

### 2.2 部署约束
* **全量名单机制：** 每台机器内置一份包含所有机位 IP、MAC 地址及主机名的静态列表。
* **隐蔽性：** 无进程界面、无任务栏图标、服务名称模拟系统核心组件。

### 3.2 鬼影模式 (Ghost Mode)
* 若决策结果为 `BLOCK`（拦截关机）：
    * **黑屏：** 通过系统 API 强制关闭显示器电源 (Power Off Monitor)、并注销所有用户会话。
    * **锁屏：** 禁用键盘鼠标输入、并卸载设备 (`BlockInput`)。
    * **静音：** 瞬间将系统音量设为 0。
    * **欺骗：** 向关机调用者返回 `STATUS_SUCCESS`，让系统认为关机成功，实则后台继续运行。

### 3.3 网络协议与协商逻辑
* **混合协议：** * **UDP 广播：** 用于瞬间宣告关机意图（`SHUTDOWN_INTENT`）。
    * **TCP P2P：** 用于关键时刻对名单内节点进行强制状态同步和存活确认。
* **并发冲突处理（关机风暴）：**
    * **随机优先级：** 每个节点发起关机请求时，随包生成一个随机浮点权重 $P$。
    * **观察窗口：** 收到本地关机指令后进入一定长时间观察期，收集全网其他节点的意图包。
    * **共识排序：** 所有节点基于收到的 $P$ 值对所有“申请关机节点”进行一致性排序。权重前 `Total - N` 名允许物理关机，其余进入鬼影模式。

### 3.4 唤醒与自救逻辑
* **意外离线判定：** 节点失联且未发出 `GOODBYE` 信号时，判定为异常关机。
* **WoL 补位：** 若存活数 $S < N$，根据名单顺序或随机策略，向离线机器发送 WoL 魔术包。
* **孤岛模式：** 若检测到存活数 $S = 1$，该节点进入“狂暴唤醒”状态，循环尝试唤醒名单中所有节点。

## 4. 非功能性需求
* **网络安全：** 所有 JSON 封包必须经过基于预设（通过配置文件设置）密钥的加密（如 AES 或简易异或混淆），防止被 Wireshark 抓包明文分析。
* **性能控制：** 协商延迟不得超过 1 秒，避免用户因等待关机时间过长而产生怀疑。
* **健壮性：** 即使网络出现瞬时波动，系统应优先选择“保活”而非放任关机。
* **CI：** 使用 github action 推送打包，调用 ./pdb 下脚本获取 RVA。
* **高度可配置性** 避免硬编码。
